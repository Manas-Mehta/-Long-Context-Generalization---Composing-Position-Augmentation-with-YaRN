## Phase 1: Data Preparation

### `scripts/prepare_data.py`

Downloads the MRCR dataset from HuggingFace, tokenizes each sample using the Qwen tokenizer, assigns samples to bins by token count, and creates train/test splits.

**Functions:**

| Function | Purpose |
|----------|---------|
| `get_bin_index(token_count)` | Maps a token count to a bin index (0=4K-8K, 1=8K-16K, etc). Returns -1 if below 4096. Uses the official MRCR bin boundaries. |
| `get_bin_label(bin_index)` | Converts bin index to human-readable label like "4K-8K". |
| `tokenize_prompt(prompt_json, tokenizer)` | Takes the MRCR `prompt` field (a JSON string of chat messages), applies the Qwen chat template, tokenizes, and returns the token count. This is the count the model will actually see at inference. |
| `main()` | Orchestrates: load tokenizer -> load dataset -> filter by needle count -> tokenize and bin each sample -> create train/test splits per bin -> save to disk. |

**Key arguments:**
- `--tokenizer` — Which tokenizer to use for binning (default: `Qwen/Qwen2.5-7B-Instruct`)
- `--output-dir` — Where to save binned data
- `--train-ratio` — Fraction for training (default: 0.7, rest is test)
- `--n-needles` — Which needle counts to include (default: `[2]` for 2-needle only)
- `--max-bin` — Highest bin to process (default: 2 = up to 16K-32K)

**Output format:** Each `train.json` / `test.json` is a list of dicts:
```json
{
  "prompt": "[{\"role\": \"user\", \"content\": \"...\"}, ...]",
  "answer": "random_string + actual_answer",
  "random_string_to_prepend": "xK7m...",
  "n_needles": 2,
  "n_chars": 150,
  "token_count_qwen": 5832,
  "bin_index": 0,
  "bin_label": "4K-8K"
}
```

**Why we re-bin with Qwen's tokenizer:** The official MRCR bins use OpenAI's `o200k_base` tokenizer. Different tokenizers produce different token counts for the same text. Since our model is Qwen, we need bins based on what Qwen actually sees.

### `hpc/prepare_data.slurm`

SLURM job for data preparation. No GPU needed (CPU-only tokenization). Requests 32G RAM, 4 CPUs, 1 hour. Does NOT set `HF_HUB_OFFLINE=1` since it needs to download the MRCR dataset.

## Phase 2: Baseline Evaluations (HIGH PRIORITY)

### `scripts/eval_mrcr.py`

Core evaluation script that supports three modes:
1. **Vanilla** — Base Qwen2.5-7B-Instruct, no modifications
2. **YaRN** — Base model with YaRN RoPE scaling enabled (zero training)
3. **LoRA** — Base model + merged LoRA adapter (for RPE experiments in Phase 3)

**Functions:**

| Function | Purpose |
|----------|---------|
| `grade_mrcr(response, answer, random_string_to_prepend)` | Official MRCR grading function. Checks that response starts with the required random prefix, strips it from both response and answer, then computes `SequenceMatcher.ratio()` (0.0 to 1.0). A score of 1.0 = perfect match. |
| `load_model(base_model_name, lora_ckpt_dir, enable_yarn, yarn_factor, ...)` | Loads the model in one of three configurations. For YaRN: injects `rope_scaling` config before loading weights. For LoRA: loads adapter with PEFT, then merges with `merge_and_unload()`. Returns `(model, tokenizer, config_description)`. |
| `evaluate_mrcr(model, tokenizer, test_data, max_new_tokens)` | Main eval loop. For each sample: parses the JSON message list -> applies chat template -> generates with greedy decoding -> grades response -> tracks per-bin scores. Returns overall score, per-bin breakdown, and all predictions. |
| `main()` | CLI entrypoint. Parses args, loads model, loads test data, runs evaluation, prints summary, saves results. |

**Key arguments:**
- `--base-model` — Model name/path (default: `Qwen/Qwen2.5-7B-Instruct`)
- `--lora-ckpt` — Path to LoRA checkpoint directory (optional, for Phase 3)
- `--enable-yarn` — Flag to activate YaRN RoPE scaling
- `--yarn-factor` — YaRN scaling factor (default: 4.0 = extends to ~131K tokens)
- `--test-file` — Path to test JSON from `prepare_data.py`
- `--output-dir` — Where to save results
- `--max-new-tokens` — Generation limit (default: 2048, MRCR answers can be long)
- `--max-examples` — Cap number of examples for quick testing (0 = all)

**Output files:**
- `eval_results.json` — Config, overall score, per-bin breakdown (no raw predictions)
- `predictions.json` — Per-example details: score, response preview, token counts, generation time

**How YaRN is enabled:**
```python
config = AutoConfig.from_pretrained(base_model_name)
config.rope_scaling = {
    "type": "yarn",
    "factor": 4.0,
    "original_max_position_embeddings": config.max_position_embeddings,  # 32768
}
model = AutoModelForCausalLM.from_pretrained(base_model_name, config=config, ...)
```
This modifies the RoPE frequency computation inside the model. The weights are unchanged. No training needed.

### `hpc/eval_vanilla.slurm`

Runs vanilla Qwen2.5-7B-Instruct on both 4K-8K and 8K-16K test bins sequentially. Uses `HF_HUB_OFFLINE=1` (model already cached from Phase 2). Requests 1 GPU, 64G RAM, 4 hours, with `--requeue` for preemption resilience.

### `hpc/eval_yarn.slurm`

Same as vanilla but passes `--enable-yarn --yarn-factor 4.0`. Identical resource requirements. The only difference is the YaRN config injection at model load time.

## MRCR Grading: How It Works

The MRCR task asks the model to reproduce a specific piece of text (a "needle") that appeared earlier in a long multi-turn conversation. To prevent models from guessing, each sample has a unique `random_string_to_prepend` that the model must output first.

**Grading logic:**
1. Check if the response starts with `random_string_to_prepend`. If not, score = 0.0 (the model didn't follow instructions).
2. Strip the random prefix from both response and expected answer.
3. Compute `SequenceMatcher(None, response, answer).ratio()` — this is a fuzzy string similarity score from 0.0 (completely different) to 1.0 (identical).

**Why SequenceMatcher, not exact match:** The expected answer can be a full paragraph. Minor formatting differences (whitespace, punctuation) shouldn't zero out the score. SequenceMatcher gives partial credit for mostly-correct responses.
